# Hippius artifact storage

RADAR's non-DB artifacts (proposals, dispatch records, frontier snapshots,
checkpoints, architecture code, training metadata) live in S3-compatible
object storage. With Phase 7 of TEN-240 these artifacts can be **dual-written
to Hippius** (Substrate-based decentralized storage) alongside the existing
R2 (or Hippius-S3) backend. Hippius is the durable, censorship-resistant
record; R2 stays as the hot-path cache.

> Dual-write is opt-in. The default deploy keeps the historical R2-only
> behaviour. Set `RADAR_DUAL_WRITE_ARTIFACTS=true` to turn it on.

## Path â†’ metadata mapping

When an artifact is written through `shared.artifact_store.ArtifactStore` it
carries a metadata dict alongside the bytes. The metadata is the equivalent
of the R2 key path â€” search-discoverable on the Hippius side once the SDK
gains tag-based listing (TEN-242). Until then the metadata travels inside
the bundle/header so a fetcher can verify what they got.

| R2 key | Hippius metadata |
|--------|------------------|
| `round_{id}/proposals/{uid}.json` | `app=radar`, `kind=proposal`, `round_id={id}`, `miner_uid={uid}` |
| `round_{id}/miner_{hk}/checkpoint.safetensors` | `app=radar`, `kind=checkpoint`, `round_id={id}`, `miner_hotkey={hk}` |
| `round_{id}/miner_{hk}/architecture.py` | `app=radar`, `kind=architecture`, `round_id={id}`, `miner_hotkey={hk}` |
| `round_{id}/miner_{hk}/training_meta.json` | `app=radar`, `kind=training_meta`, `round_id={id}`, `miner_hotkey={hk}` |
| `round_{id}/miner_{hk}/stdout.log` | `app=radar`, `kind=stdout`, `round_id={id}`, `miner_hotkey={hk}` |
| `round_{id}/dispatch/vali_{hk}.json` | `app=radar`, `kind=dispatch`, `round_id={id}`, `validator_hotkey={hk}` |
| `frontier/{task}/latest.json` | `app=radar`, `kind=frontier`, `task={task}` |

`app=radar` is the global tag that scopes RADAR's writes against any other
tenant on the same Hippius account.

## Substrate audit-trail integration

When a validator writes an artifact through `ArtifactStore`, the returned
`hippius_cid` is appended to that experiment's `substrate_cids` JSONB
column (see Phase 5):

```json
[
  {"kind": "phase_c_record", "validator_hotkey": "5G...", "cid": "bafy1..."},
  {"kind": "checkpoint",     "cid": "bafy2..."},
  {"kind": "architecture",   "cid": "bafy3..."},
  {"kind": "training_meta",  "cid": "bafy4..."}
]
```

The verifier (`tools/verify_radar_experiment.py`) already iterates this
list and inspects each entry; new `kind` values surface as additional
verifications without a tool-side change.

## What's wired today

Validator-side, dual-writeable as of this PR:

* Dispatch records (`coordinator.write_dispatch_record`)
* Frontier snapshots (`coordinator.write_frontier`)
* Training metadata cached during Phase B (`coordinator._cache_training_meta`,
  whenever the validator pulls a meta back from R2 to seed the DB cache)

Validator-side, **deferred** until trainer-side credentials land:

* **Checkpoints.** The trainer subprocess uploads to R2 via presigned URLs
  generated by the validator. Routing checkpoints through Hippius requires
  either trainer-side Hippius creds (operationally invasive) or a
  validator-side post-fetch dual-write (extra bandwidth on every dispatcher).
  Both are real options; the cost note below frames the decision.
* **Architecture code, stdout logs.** Same flow as checkpoints â€” uploaded
  by the trainer via presigned URLs.

The trainer-side flow can be migrated piecemeal: each call site that
currently does `r2.upload_text(...)` is one-line-replaceable with
`artifact_store.put_bytes(...)` once the trainer holds a `HippiusClient`.

Agent scratchpad (miner-facing presigned URLs) stays R2-only by design â€”
that surface is short-lived and miner-controlled.

## Rollout sequence

1. Merge with `RADAR_DUAL_WRITE_ARTIFACTS=false` everywhere â€” no change.
2. Enable in staging for ~1 week. Watch the dual-write counters
   (`artifact_store.dual_write.r2_ok`, `.hippius_ok`, `.both_failed`,
   `.bytes_written`) and the Hippius bill.
3. Enable in production validators for a month. Same observation set.
4. **Separate decision**: flip read preference to Hippius-first for
   specific kinds (frontier first â€” small, infrequent, auditable),
   keep checkpoints as R2-primary with Hippius cold-storage.
5. Eventually deprecate R2 paths kind by kind as confidence grows.

No deprecation in this PR. Both R2 paths and Hippius writes remain
available indefinitely; operators choose what to enable.
